# Claude Code PR 处理规范

> 本文件用于指导 Claude Code 在本仓库中提交和处理 Pull Request。每次新增或修改功能时，请严格遵循以下规则。

## 一、核心原则

1. **一切新功能都通过 PR 提交**，不要直接向主分支（main / master）推送代码。
2. **每个 PR 只做一件事**：只实现或修改单一功能。
3. **PR 要小、粒度要细**：优先小步提交；遇到大功能，先拆分成多个相互独立的 PR 分步提交，而不是一次性提交一个庞大的改动。
4. **合并后主分支必须始终可运行**：任何人在任意时间检出主分支，都应能复现演示效果。

## 二、标准工作流程

Claude Code 在收到一个功能需求时，按以下步骤操作：

1. 先判断该需求是否能拆分。若涉及多个独立功能点，先告知用户拆分方案，按功能逐个开 PR。
2. 从最新的主分支切出一个新分支，分支名能体现本次改动，例如 `feature/user-login`、`fix/null-pointer`。
3. 在该分支上只实现**这一个**功能，不夹带无关改动（无关的格式调整、重构、依赖升级都应另开 PR）。
4. 提交前在本地完成自测，确认功能正常、主流程可运行。
5. 按下方模板撰写 PR 标题与描述。
6. 推送分支并创建 PR，等待评审。

## 三、PR 标题与描述模板

每个 PR 的标题和描述都必须**清晰、完整**，缺一不可。直接套用下面的模板：

### 标题
用一句话说明本 PR **新增 / 修改**了什么。

> 示例：`feat: 新增用户邮箱验证码登录`

### 描述正文

```markdown
## 功能描述
说明该功能的作用，以及使用方式（谁会用、怎么用、入口在哪）。

## 实现思路
简要说明技术选型或核心实现逻辑（用了什么方案、关键改动在哪、为什么这么做）。

## 测试方式
说明如何验证该功能正常运行：
- 复现 / 验证步骤
- 预期结果
- 涉及的测试命令或测试用例（如有）
```

## 四、提交前自检清单

创建 PR 之前，Claude Code 应逐项确认：

- [ ] 本 PR 只对应**一个**功能或修复，没有夹带无关改动。
- [ ] 改动足够小、足够聚焦；大功能已被拆分为多个独立 PR。
- [ ] 标题用一句话说清了「新增 / 修改了什么」。
- [ ] 描述包含**功能描述**、**实现思路**、**测试方式**三部分，且内容完整。
- [ ] 已在本地自测通过，给出了可复现的验证步骤。
- [ ] 合并这个 PR 后，主分支代码仍能正常运行、能复现演示效果。

## 五、Claude Code 行为要求

- 当用户的请求实际上包含多个功能时，**主动提示拆分**，不要把它们塞进同一个 PR。
- 不要为了「省事」而合并多个改动；宁可多开几个小 PR。
- 在准备提交涉及推送分支、创建 / 修改 PR 等操作前，先向用户说明将要执行的动作并取得确认，再执行。
- 写完描述后自查是否齐全（功能描述 / 实现思路 / 测试方式），缺项要补齐。
